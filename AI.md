# Maximizing AI Assistant Usage Credits

This guide helps you get more value from usage-limited AI assistants such as Codex, Claude, Antigravity, ChatGPT, Cursor-style agents, and similar coding or productivity tools.

## Core Principle

AI credits are usually spent on three things:

- Context read by the model
- Tokens generated by the model
- Tool work such as terminal commands, file reads, browser actions, web search, or long-running agents

The goal is not to use the fewest messages. The goal is to reduce wasted context, vague retries, repeated scans, and unnecessary output.

## Start With A Tight Task

Good:

```text
Fix the login redirect bug. Read Auth.tsx, App.tsx, and userService.ts first. After the fix, run npm run build.
```

Expensive:

```text
Check my whole app and fix anything wrong.
```

Use this structure:

```text
Goal:
Files/areas to inspect:
What to change:
What not to change:
How to verify:
```

Example:

```text
Goal: Move event image storage from Firestore Base64 to Sanity URLs.
Files/areas to inspect: components/CreateEventForm.tsx, services/eventService.ts, services/imageUtils.ts.
What to change: Upload Base64 images to Sanity before Firestore save.
What not to change: Do not refactor event notifications or UI layout.
How to verify: npm run build.
```

## Give The Assistant The Right Context

Prefer giving:

- Exact file names
- Error messages
- The command that failed
- The desired behavior
- The actual behavior
- Any constraints, such as "do not change the design" or "no new dependencies"

Avoid giving:

- Huge pasted files if the assistant can read the repo
- Screenshots of code when text is available
- Multiple unrelated goals in one prompt
- Vague phrases like "make it better" without criteria

## Batch Related Work

Each task switch costs context. Group related changes together.

Better:

```text
Implement Sanity image upload, add env typings, update the form submit flow, and run the build.
```

Worse:

```text
Add Sanity.
Now add env.
Now update the form.
Now build it.
```

But do not batch unrelated work:

```text
Fix image upload, redesign the homepage, update Firebase rules, and create a marketing plan.
```

That usually causes expensive context drift.

## Ask For Diffs, Not Essays

When you want code changes, ask the assistant to implement and summarize briefly.

Good:

```text
Make the change. Keep the final summary short: changed files, verification, and any blockers.
```

Expensive:

```text
Explain every line and every possible alternative before coding.
```

Use deep explanations only when you actually need to learn or review architecture.

## Let The Assistant Use The Repo

For coding agents, do not paste files that already exist in the workspace unless needed. Say:

```text
Please scan the relevant files before editing.
```

This is usually cheaper and safer than pasting thousands of lines into chat, especially when the agent has file tools.

## Limit Broad Searches

Broad instructions burn credits:

```text
Search the whole repo and find every issue.
```

Better:

```text
Search for where event imageUrl is written or rendered, then update only the save path.
```

Good search targets:

- Function names
- Component names
- Error strings
- API route names
- Database field names
- Console error snippets

## Prefer One Verification Command

Tell the assistant what verification matters most.

Examples:

```text
Run npm run build.
Run npm run lint only.
Run the specific test file for auth.
Do not run the full test suite unless needed.
```

Full test suites, browser automation, and builds are valuable, but they can be expensive. Use them when the change risk justifies it.

## Avoid Repeating Full Context

If you already gave context in the same thread, refer to it:

```text
Continue from the Sanity migration. Now add a small admin view for uploaded images.
```

Do not paste the whole previous plan again unless the thread lost context.

## Use Checkpoints For Big Work

For large tasks, ask for staged execution:

```text
First inspect and propose a short plan. Then implement after that unless you find a blocker.
```

Or:

```text
Make the smallest safe fix first. Avoid broad refactors.
```

This prevents the assistant from spending credits building the wrong thing.

## Control Output Size

Use clear output limits:

```text
Keep the final answer under 10 bullets.
Do not paste full files.
Only show changed code snippets.
Summarize errors instead of dumping logs.
```

Large answers cost credits and are often harder to use.

## Be Explicit About Autonomy

When you want action:

```text
Implement it directly.
```

When you only want advice:

```text
Do not edit files yet. Just review and explain options.
```

When you want safety:

```text
Ask before installing dependencies or changing config.
```

This avoids wasted back-and-forth.

## Reuse Existing Patterns

Tell the assistant to follow local conventions:

```text
Use existing service patterns. Do not introduce a new state library.
Match the current component style.
Keep the change scoped.
```

This reduces exploratory code and cleanup.

## For Debugging

Provide this template:

```text
Command:
Expected:
Actual:
Error:
Recent changes:
Relevant files:
```

Example:

```text
Command: npm run build
Expected: production build succeeds
Actual: Vite fails
Error: Could not resolve sanity.config.ts
Recent changes: added Sanity client
Relevant files: package.json, vite.config.ts, sanity.config.ts
```

## For UI Work

Give the target screen and interaction:

```text
Update the Create Event image upload area. Keep the existing layout. Add upload progress while Sanity upload is running. Verify desktop and mobile.
```

Avoid:

```text
Make the UI nicer.
```

UI agents can spend a lot of credits exploring taste. Give constraints.

## For Research

Ask for sourced, focused research:

```text
Check the current official Sanity docs for browser image upload limitations. Summarize only what affects this implementation.
```

Avoid:

```text
Research everything about Sanity.
```

## For Code Review

Use:

```text
Review only the Sanity migration diff. Prioritize bugs, security risks, and missing verification. Do not comment on unrelated files.
```

This keeps the review from ballooning.

## Credit-Saving Prompt Examples

### Implement A Feature

```text
Implement [feature]. Read [files] first. Keep changes scoped to [area]. Do not change [constraints]. Verify with [command]. Final summary only: changed files, verification, blockers.
```

### Fix A Bug

```text
Fix this bug:
Expected:
Actual:
Error:
Relevant files:
Please inspect before editing, make the smallest safe fix, and run [verification].
```

### Explain Code

```text
Explain how [feature/function] works. Use file references. Keep it concise and focus on the execution flow.
```

### Refactor

```text
Refactor [specific area] to [goal]. Preserve behavior. Do not rename public APIs unless necessary. Run [tests/build].
```

## Things That Waste Credits

- Asking for "full code" when only a small patch is needed
- Pasting files the assistant can already read
- Switching topics in the same thread repeatedly
- Asking for broad rewrites without constraints
- Requesting exhaustive explanations before every small change
- Running full test suites after tiny isolated changes
- Asking multiple assistants the same vague task instead of giving one assistant a precise brief

## Things That Save Credits

- Naming exact files and commands
- Giving expected vs actual behavior
- Setting boundaries
- Asking for short summaries
- Letting coding agents inspect files directly
- Verifying with the smallest meaningful command
- Keeping one thread focused on one project goal

## Best Default Prompt

```text
Please implement this directly.

Goal:
Relevant files:
Constraints:
Verification:

Before editing, inspect the relevant files. Keep the change scoped. In the final answer, only include changed files, verification result, and any blockers.
```
